# 売上ダッシュボード 課題整理と修正方針

## 1. 前提の整理

- **販売形態**: 出品者（パートナー組織）単位で出品・売上が発生する。
- **データ上の「出品者」**:
  - **users.id** … 個人ユーザー（商品の `products.seller_id` はここを参照）
  - **partners.id** … 出品者としての組織（1 組織に複数ユーザーが属しうる）
- **注文データ**:
  - **order_items.seller_id** … `users.id`（その明細の商品を出品したユーザー）
  - **orders.seller_id** … `partners.id`（マイグレーション 007 で「先頭の order_item の出品ユーザーの partner_id」を格納）
- **売上台帳（ledger）**: `partner_id`（partners.id）単位で売上・手数料・送金を管理。

したがって、「出品者単位の売上」＝**パートナー（partners.id）単位**で集計する必要がある。

---

## 2. 現状の課題

### 2.1 集計軸の不整合（最重要）

| 箇所 | 現状の絞り込み | 問題 |
|------|----------------|------|
| ダッシュボード・売上合計 | `oi.seller_id = uid OR o.seller_id = partner_id` | `order_items.seller_id` は users.id のため `oi.seller_id = partner_id` はほぼマッチしない。実質「ログインユーザー本人の明細」のみ集計されている。 |
| 取引一覧・件数 | 同上 | 同上。パートナー全体の取引になっていない。 |
| getRevenueCardData（カードチャート） | `oi.seller_id = $2`（uid のみ使用） | 第3引数 `partner_id` を渡しているが関数内で未使用。**パートナー単位の売上になっていない。** |
| buildSellerFilters / getAnalyticsBucketsV2（詳細分析） | `oi.seller_id = $1`（uid） | 詳細ページもユーザー単位のまま。 |

結果として、**同じパートナーに属する別ユーザーの売上・取引が含まれず、「出品者単位」と一致していない。**

### 2.2 売上金額の定義の曖昧さ

- **ダッシュボードで表示している額**: `SUM(oi.price * oi.quantity)`（注文時点の商品合計、税・手数料なし）
- **台帳（ledger）**: 注文総額ベースで手数料 6% を控除し、`partner_id` 単位で「売上」「手数料」「送金」を管理。

そのため、

- 表示されている「売上」が**手数料控除前**なのか**控除後（手取り）**なのかが不明瞭
- 台帳と連動しておらず、**送金予定・入金済み**などの情報がダッシュボードに出てこない

という状態になっている。

### 2.3 出品者ダッシュボードで不足している情報

- 手数料控除後の**手取り額**（または控除前/控除後の切り替え）
- **送金状況**（未送金・送金予定日・入金済みなど）
- 商品別・カテゴリ別の売上内訳（どの商品がどれだけ売れているか）
- 期間比較（前月比・前年比など）や、注文数・平均単価などの KPI

現状は「売上推移チャート＋注文数＋直近取引リスト」が中心で、**売上金額を確認する画面として十分でない**という指摘と整合している。

### 2.4 その他の不整合

- **getRevenueCardData**: 呼び出しは `getRevenueCardData(dbQuery, uid, partner_id)` の 3 引数だが、定義は `(dbQuery, sellerId)` の 2 引数のみ。`partner_id` が無視されている。
- 取引一覧（`/seller/trades`）の WHERE は `oi.seller_id = $1`（uid）のみで、管理者用の `owner_all` 以外はパートナー単位になっていない。

---

## 3. 修正方針

### 3.1 集計軸の統一（パートナー単位）

**方針**: 出品者ダッシュボードおよび売上詳細は、**「現在ログインしているユーザーが属する partner_id」** を基準に集計する。

- **パートナーに属さないユーザー**（`partner_id` が NULL）の場合は、従来どおり **users.id（本人）** のみを対象にする。
- 集計クエリの変更イメージ:
  - 現状: `WHERE oi.seller_id = $uid`
  - 変更:  
    `JOIN users seller_user ON seller_user.id = oi.seller_id`  
    `WHERE (seller_user.partner_id = $partner_id OR (seller_user.partner_id IS NULL AND oi.seller_id = $uid))`  
    または、パートナー所属時は `seller_user.partner_id = $partner_id` のみでよい。

これにより、

- ダッシュボードの「売上合計」
- 取引件数・直近取引リスト
- getRevenueCardData（今週/今月/全期間のチャート）
- buildSellerFilters / getAnalyticsBucketsV2（詳細分析）

のすべてで、**出品者（パートナー）単位の売上**が一貫して表示されるようにする。

### 3.2 売上金額の定義の明確化

- **表示する「売上」の定義を 2 つに整理する**（実装は段階でよい）:
  1. **売上（税抜・手数料控除前）**: 現在の `SUM(oi.price * oi.quantity)` をそのまま「商品売上」として表示。
  2. **手取り（想定）**: 上記からプラットフォーム手数料（例: 6%）を差し引いた額。  
     可能であれば、**ledger の type=sale / platform_fee を集計**して「確定した手取り」を表示すると、台帳と一致する。

- ラベルとヘルプで「売上（手数料控除前）」「手取り（手数料控除後）」を明示する。

### 3.3 ダッシュボード UI の改善

- **KPI の明確化**
  - 今月（または選択期間）の「売上（円）」「注文数」「平均注文単価」を目立つ形で表示。
  - 手数料控除後の「手取り」を表示する場合は、別枠またはタブで区切る。

- **売上カード**
  - データソースを上記のパートナー単位集計に差し替え（getRevenueCardData の修正と引数統一）。
  - 表示内容は「今週/今月/全期間」のままでもよいが、集計軸がパートナー単位であることを説明する文言を追加してもよい。

- **取引履歴**
  - 既存の「直近 N 件」「すべて見る」は維持しつつ、取得クエリをパートナー単位に変更。

- **売上詳細（/seller/analytics）**
  - 期間・粒度・フィルタは現状のまま、集計クエリのみパートナー単位に変更。
  - 余力があれば「商品別売上」「カテゴリ別」などの内訳テーブルを追加すると、売上確認としてさらに使いやすくなる。

### 3.4 台帳との連携（任意・段階的）

- 手数料や送金状況を表示する場合は、**ledger テーブル**を参照する。
  - 例: `partner_id` ごとの `type IN ('sale','platform_fee','payout')` の集計で「売上」「手数料」「送金済み」を表示。
- まずは「売上（控除前）」と「手取り（控除後）」の説明と、必要なら手数料率の表示にとどめ、送金状況の表示は二期で実装してもよい。

### 3.5 管理者（owner_all）の扱い

- 管理者が「すべての売上」を見る場合は、現状どおり `owner === 'owner_all'` でフィルタをかけない（全 order_items 対象）でよい。
- その際も、集計は「注文・明細」ベースで一貫させる（ORDER 単位の件数と、order_items の売上合計の関係を明確にする）。

## 4. 実装時の修正箇所一覧

| 優先度 | 対象 | 修正内容 |
|--------|------|----------|
| 高 | `getRevenueCardData` | 第2引数を `partner_id` に変更（または uid と partner_id の両方を受け、partner_id があればパートナー単位で集計）。SQL を `users.partner_id` 経由の絞り込みに変更。 |
| 高 | `/dashboard/seller` の売上合計クエリ | 集計をパートナー単位に変更（`JOIN users ON users.id = oi.seller_id` の上で `users.partner_id = $2`、パートナー未所属時は `oi.seller_id = $1`）。 |
| 高 | `/dashboard/seller` の取引件数・直近取引クエリ | 上記と同様に、パートナー単位で取得する WHERE に変更。 |
| 高 | `buildSellerFilters` | 通常時（owner !== 'owner_all'）の条件を、パートナー単位に変更。`sellerId` の代わりに `partnerId` を受け、`EXISTS (SELECT 1 FROM order_items oi JOIN users u ON u.id = oi.seller_id WHERE oi.order_id = o.id AND u.partner_id = $1)` のような形にする。 |
| 高 | `getAnalyticsBucketsV2` | `buildSellerFilters` に渡す引数を `partner_id` ベースに変更（呼び出し元で partner_id を渡す）。 |
| 高 | `/seller/analytics` のルート | 現在の `uid` に加え、ユーザーの `partner_id` を取得し、`getAnalyticsBucketsV2` および `buildSellerFilters` にパートナー単位で渡す。 |
| 高 | `/seller/trades` の WHERE | 通常時をパートナー単位に変更（`oi.seller_id = uid` ではなく、`oi` と `users` を JOIN して `users.partner_id = partner_id` で絞る）。 |
| 中 | 売上カード・詳細ページのラベル | 「売上（手数料控除前）」「手取り」などの説明を追加。 |
| 中 | ダッシュボード KPI | 今月売上・注文数・平均単価などをカードで表示（既存レイアウトに組み込む）。 |
| 低 | ledger 連携 | 手数料控除後・送金状況の表示（二期）。 |

---

## 5. まとめ

- **課題の本質**: 売上・取引の集計が「ユーザー（uid）単位」になっており、「出品者（パートナー）単位」になっていない。加えて、売上金額の定義が曖昧で、台帳と連携していない。
- **修正の核**: 集計軸を **partner_id** に統一し、`order_items` は `users.id` 経由で `users.partner_id` と紐付けて集計する。
- **UI**: 集計軸を直したうえで、売上・手取り・注文数などの KPI を明確にし、必要に応じて台帳と連携して「本当に使える」売上ダッシュボードに仕立てる。

この方針に沿って、まずは「集計軸の統一」から順に実装することを推奨する。
